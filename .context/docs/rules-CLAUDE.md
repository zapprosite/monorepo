# Claude Monorepo Development Guide (CLAUDE.md)

This document outlines the architectural standards, coding patterns, and workflow requirements for the monorepo. It serves as the authoritative technical specification for developers and AI assistants to ensure consistency across backend services, frontend applications, and shared packages.

---

## 🏗 Monorepo Architecture

The repository is a **Turborepo** monorepo designed for high-performance TypeScript applications with full end-to-end type safety.

### Technology Stack
- **Backend**: Node.js, Fastify, [Orchid ORM](https://orchid-orm.netlify.app/), tRPC, OpenTelemetry.
- **Frontend**: React 19, Vite, TanStack Query, tRPC Client, Material UI (MUI).
- **Database**: PostgreSQL (managed via Orchid ORM using `snake_case` conventions).
- **Shared**: Zod schemas for validation and shared UI component libraries.
- **Package Management**: Yarn (v1.22.22).

---

## 🛠 Core Tooling & Commands

### Development Operations
| Command | Action |
| :--- | :--- |
| `yarn dev` | Launches all applications (Frontend, API, AI Gateway) in parallel. |
| `yarn build` | Builds all workspaces according to the Turbo dependency graph. |
| `yarn check-types` | Executes `tsc` across all packages to verify type safety. |

### Code Quality
- **Linting & Formatting**: The project uses **Biome** for lightning-fast linting and formatting. Run `yarn format` to fix style issues and `yarn lint` to check for errors.

### Database Management
Database operations are centered in `apps/api` using Orchid ORM:
- **Generate Migration**: `yarn run db g <migration_name>`
- **Apply Migrations**: `yarn run db up`
- **Rollback**: `yarn run db down`

## 📐 Architectural Principles

### 1. The "No Barrel Export" Rule (`packages/`)
To ensure optimal build times and efficient tree-shaking (preventing large frontend bundles), shared packages **must not** use barrel exports (`index.ts` files that re-export everything).

- ✅ **Correct**: `import { UserCreateInput } from '@repo/zod-schemas/user.zod'`
- ❌ **Incorrect**: `import { UserCreateInput } from '@repo/zod-schemas'`

### 2. Schema-Driven Development (`packages/zod-schemas`)
The source of truth for all data structures is `packages/zod-schemas`. Every entity follows a strict structural pattern to handle different lifecycle stages:

| Schema Suffix | Purpose |
| :--- | :--- |
| `Mandatory` | Core required fields for the entity. |
| `Optional` | Partial or nullable fields. |
| `AutoGenerated` | Fields handled by DB (e.g., `id`, `createdAt`, `updatedAt`). |
| `CreateInput` | Used for POST/Mutation operations (excludes `AutoGenerated`). |
| `UpdateInput` | All fields are partial; used for PATCH operations. |
| `SelectAll` | The complete object shape returned by the API. |

### 3. Backend Implementation (`apps/api`)
- **Table Definitions**: Located in `src/modules/<feature>/tables/`. Use `BaseTable` to ensure consistent `snake_case` mapping to Postgres.
- **tRPC Procedures**:
    - `publicProcedure`: Unauthenticated access.
    - `protectedProcedure`: Requires a valid session.
- **Registration**: All new tables must be registered in `apps/api/src/db/db.ts` to be accessible via the ORM.

### 4. Frontend Implementation (`apps/web`)
- **End-to-End Safety**: The frontend imports the `AppTrpcRouter` type directly from the backend. This allows for full IDE autocompletion for API calls without a code-generation step.
- **Hooks**: Use `trpc.<module>.<procedure>.useQuery()` for data fetching and `trpc.<module>.<procedure>.useMutation()` for state changes.

## 🔒 Security & Error Handling

### Centralized Error Parsing
The backend utilizes `apps/api/src/utils/errorParser.ts` to automatically map internal errors to HTTP responses:
- **Zod Validation Errors**: Returns `400 Bad Request` with structured field errors.
- **Unique Constraint Violations**: Returns `409 Conflict`.
- **Foreign Key Failures**: Returns `400 Bad Request`.

### Session Security
Sessions are managed via `DatabaseSessionStore`. On every request, the system:
1. Validates the session token in the database.
2. Checks the user agent and IP address to detect session hijacking.
3. Levels of security can be enforced using `SessionSecurityLevel`.

## 🚫 Common Pitfalls to Avoid

1. **Generic IDs**: Do not name columns simply `id`. Always use entity-prefixed names like `clientId`, `userId`, or `orderId`.
2. **Type Assertions**: Strictly avoid `as any`. Use the shared Zod schemas to safely validate or cast data.
3. **Manual SQL**: Avoid raw SQL queries. Use Orchid ORM's type-safe query builder to ensure database changes don't break the application.
4. **Missing Migrations**: If you modify a table definition in TypeScript, you **must** generate and run a migration (`yarn run db g`) to sync the physical database.
