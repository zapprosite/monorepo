# Claude Monorepo Development Guide

This documentation serves as the comprehensive technical standard for the monorepo. It details the architecture, development workflows, and implementation patterns required to maintain consistency across the React frontend, Fastify/tRPC backend, and shared TypeScript packages.

## 🏗 System Architecture

The repository is a type-safe full-stack ecosystem managed by **Turborepo**. It leverages a "Schema-First" philosophy where Zod definitions drive both database constraints and API types.

### Technical Stack
| Layer | Technology |
| :--- | :--- |
| **Monorepo Management** | Turborepo + Yarn Workspaces |
| **Backend** | Fastify (Node.js) |
| **API Protocol** | tRPC (End-to-end type safety) |
| **Database/ORM** | PostgreSQL + [Orchid ORM](https://orchid-orm.netlify.app/) |
| **Frontend** | React 19 + Vite + TypeScript |
| **UI Components** | Shared MUI + Custom library in `packages/ui` |
| **Validation** | Centralized Zod (in `packages/zod-schemas`) |
| **Formatting/Linting** | Biome |

## 📐 Core Implementation Standards

### 1. Schema-First Development (`packages/zod-schemas`)
Every new feature must begin with a Zod schema. These schemas generate the TypeScript types used by the ORM and tRPC.

**Schema Naming Convention:**
- `Mandatory`: Common core fields.
- `AutoGenerated`: Fields set by DB (e.g., `id`, `createdAt`, `updatedAt`).
- `CreateInput`: `Mandatory` fields minus `AutoGenerated`.
- `UpdateInput`: A partial version of `CreateInput`.
- `SelectAll`: The final validated shape for API responses.

**Example usage in code:**
```typescript
// Always use deep imports to keep bundles lean
import { UserSelectAll } from '@repo/zod-schemas/user.zod';
```

### 2. Backend Logic (`apps/api`)
- **Table Definitions**: Located in `src/modules/<feature>/tables/`. Use `BaseTable` to ensure consistent `snake_case` mapping.
- **ORM Registration**: New tables must be registered in `apps/api/src/db/db.ts`.
- **Procedures**:
  - `publicProcedure`: Open access.
  - `protectedProcedure`: Requires a session verified against `SessionTable`.

### 3. Frontend Data Flow (`apps/web`)
The frontend consumes the `AppTrpcRouter` type. No manual Type/DTO definition is required on the client.

```typescript
// Pattern for Hook Usage
const { data, isLoading } = trpc.clients.getById.useQuery({ clientId: '...' });

const mutation = trpc.clients.create.useMutation({
  onSuccess: () => utils.clients.list.invalidate()
});
```

## 🛠 Development Workflow

### Essential Commands
- `yarn dev`: Starts all applications (Web, API, AI Gateway) concurrently.
- `yarn build`: Optimized production build via Turbo.
- `yarn format`: Runs Biome to fix style/linting issues.
- `yarn check-types`: Validates TypeScript across all workspaces.

### Database Management (Orchid ORM)
All commands run from the `apps/api` directory:
- **Generate Migration**: `yarn run db g <description_name>`
- **Apply Migrations**: `yarn run db up`
- **Rollback**: `yarn run db down`
- **Seed Database**: `yarn run db seed`

## 🔒 Security & Reliability

### Authentication & Sessions
- **Session Security**: The system uses `sessionSecurity.middleware.ts` which validates the session ID, compares `UserAgent`, and verifies the `IP Address` to prevent session hijacking.
- **RBAC**: User permissions are enforced via the `UserRolesTable`.

### Error Handling
The backend uses a standardized `AppError` class.
- **400**: Zod Validation error.
- **404**: Resource not found.
- **409**: Database Conflict (Unique constraint violation).
- **500**: Unexpected Internal Error.

## 🚫 Development Constraints

1. **No Barrel Exports**: Do not use `index.ts` files to export entire directories in shared packages. This avoids bloated frontend bundles and tree-shaking failures.
2. **Explicit IDs**: Avoid generic `id` column names in the DB. Use `teamId`, `userId`, etc., to prevent ambiguity in complex joins.
3. **No `as any`**: Use Zod's `.parse()` or `.safeParse()` to move from unknown types to validated types.
4. **ORM Only**: Raw SQL should be avoided. Use the Orchid ORM query builder to maintain type safety and prevent injection.
5. **Component Design**: UI components in `packages/ui` should be "dumb" (presentational) and accept props for behavior, keeping business logic in the `apps/` layer.
