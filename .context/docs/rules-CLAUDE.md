# Claude Code Development Guide (CLAUDE.md)

This document serves as the primary technical specification and rulebook for **Claude Code** (and other AI assistants) when operating within this monorepo. It defines the architectural patterns, coding standards, and workflow requirements to ensure consistency across the backend, frontend, and shared packages.

## Monorepo Overview

The project is a **Turborepo** monorepo designed for high-performance TypeScript applications.

*   **Backend**: Node.js, Fastify, Orchid ORM, tRPC, OpenTelemetry.
*   **Frontend**: React 19, Vite, TanStack Query, tRPC Client.
*   **Database**: PostgreSQL managed via Orchid ORM (snake_case convention).
*   **Shared**: Zod schemas for validation and UI components (MUI-based).
*   **Package Manager**: Yarn (v1.22.22).

---

## Core Tooling & Commands

### Development & Build
*   `yarn dev`: Starts all applications (frontend + backend) in development mode.
*   `yarn build`: Builds all workspaces using Turbo's dependency graph.
*   `yarn check-types`: Runs `tsc` across all packages to verify type safety.

### Code Quality
*   `yarn lint`: Executes linters across the monorepo.
*   `yarn format`: Formats code using **Biome**.

### Database Operations
Managed via Orchid ORM within `apps/api` (formerly `apps/backend`):
*   `yarn run db g <name>`: Generate a new migration.
*   `yarn run db up`: Apply pending migrations.

## Architectural Principles

### 1. Package Strategy (`packages/`)
To optimize build times and tree-shaking, all shared packages follow a **No Barrel Export** rule.

*   **Direct Imports Required**: Do not import from the package root. Import specific files.
    *   ✅ `import { userCreateInput } from '@repo/zod-schemas/user.zod'`
    *   ❌ `import { userCreateInput } from '@repo/zod-schemas'`
*   **Tree-Shaking**: Every component or utility must be independently exportable to prevent bloating the frontend bundle.

### 2. Schema-Driven Development (`zod-schemas`)
The source of truth for all data structures is `packages/zod-schemas`. 

**The Zod Pattern**:
Every entity must define specific schema variations:
*   **Mandatory**: Core required fields.
*   **Optional**: Partial or nullable fields.
*   **AutoGenerated**: Fields handled by DB (IDs, `createdAt`).
*   **CreateInput**: Used for POST/Mutation operations.
*   **UpdateInput**: All fields partial for PATCH operations.
*   **SelectAll**: The complete object shape returned by queries.

### 3. Backend Implementation (`apps/api`)
The backend uses **Fastify** and **tRPC** for a type-safe API.

*   **Tables**: Define in `src/modules/<feature>/tables/`. Use `BaseTable` to ensure `snake_case` mapping.
*   **Procedures**: 
    *   `publicProcedure`: For unauthenticated access.
    *   `protectedProcedure`: For authenticated routes. Always wrap DB logic in these to leverage the centralized error parser.
*   **Sessions**: Managed via `DatabaseSessionStore`. Sessions track device fingerprints, IP addresses, and user agents for security audits.

### 4. Frontend Implementation (`apps/web`)
The frontend uses **React 19** and the **tRPC Proxy Client**.

*   **Type Safety**: The frontend imports the `AppTrpcRouter` type directly from the backend. This provides full E2E autocompletion without code generation steps.
*   **Hooks**: Use `trpc.<module>.<procedure>.useQuery()` or `useMutation()`.
*   **Layout**: Global state and theme are handled via `AppLayout.tsx` and `ThemeProvider`.

## Database & Model Rules

When adding a new table:
1.  **Naming**: Use descriptive ID names (e.g., `clientId` instead of `id`).
2.  **Foreign Keys**: Use descriptive naming (e.g., `authorUserId`).
3.  **Registration**: New tables must be registered in `apps/api/src/db/db.ts` to be accessible via the Orchid ORM instance.

---

## Error Handling & Security

### Centralized Error Parsing
The backend utilizes a specialized `errorParser.ts`. It automatically converts:
*   Zod errors → 400 Bad Request (with field details).
*   Unique Constraint → 409 Conflict.
*   Foreign Key Violation → 400 Bad Request.

### Security Features
*   **Rate Limiting**: Configured per team in the API gateway.
*   **IP Whitelisting**: Managed via `ip-whitelist.middleware.ts`.
*   **Device Tracking**: Every session creation hashes the User Agent and headers to detect session hijacking or unusual activity.

## Environment Variables
Always refer to `.env.example` files in the root and sub-apps.
*   `DATABASE_URL`: Postgres connection string.
*   `VITE_API_URL`: Backend URL for the frontend.
*   `COOKIE_SECRET`: Used for session signing.

## Common Pitfalls to Avoid
1.  **Barrel Exports**: Never create `index.ts` files that re-export an entire directory in a package.
2.  **Type Assertions**: Avoid `as any` or `as unknown`. Use the shared Zod schemas to cast/validate data.
3.  **Generic IDs**: Always prefix IDs with the entity name (e.g., `userId`).
4.  **Missing Migrations**: Always check if a schema change in Orchid ORM requires a physical database migration.
