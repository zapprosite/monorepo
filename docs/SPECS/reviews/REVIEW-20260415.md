### Important

1. **`setup-vl-model.sh` — Hardcoded user home path**
   - File: `scripts/setup-vl-model.sh` (line ~160)
   - Code: `if [[ -f "/home/will/.hermes/config.yaml" ]]; then`
   - Issue: User-specific path bypasses env var canonical pattern. If run on a different host or as a different user, this silently skips the Hermes config update.
   - Suggestion: Use `HERMES_CONFIG="${HERMES_CONFIG:-$HOME/.hermes/config.yaml}"` and document the env var.

2. **`setup-whisper-medium-ptbr.sh` — Python HTTP server binds 0.0.0.0**
   - File: `scripts/setup-whisper-medium-ptbr.sh` (server script at `/tmp/whisper-medium-server.py`)
   - Code: `HTTPServer(('0.0.0.0', PORT), Handler)`
   - Issue: Whisper Medium PT-BR server (:8204) is exposed on all interfaces. If the host is multi-tenant or accessible beyond localhost, audio data is sent over the network unencrypted.
   - However: This is an internal-only service. Document clearly that it must not be exposed publicly. Consider binding to `127.0.0.1` if Ollama is also localhost-only.
   - Severity: Low (internal network only assumed)

3. **`audio-transcriptions.ts` — Multipart parser backtracking risk**
   - File: `apps/ai-gateway/src/routes/audio-transcriptions.ts` (line ~117)
   - Code: `const bodyStr = body.toString('binary'); const parts = bodyStr.split(sep);`
   - Issue: If a malicious multipart body contains an extremely long boundary string without actual separator matches, `split()` will allocate a large string copy. Combined with the 50MB body limit, a crafted request could cause memory pressure.
   - This is theoretical for internal services; no practical exploit path given the 50MB limit.
   - Suggestion: Add a sanity check on boundary length (e.g., max 256 chars).

4. **`AGENTS.md` / `CLAUDE.md` — Documentation inconsistency on vision model**
   - Files: `AGENTS.md`, `CLAUDE.md`, `docker-compose.yml`
   - The `docker-compose.yml` healthcheck still references `qwen2.5vl` (not updated), while all TypeScript files correctly reference `llava-phi3`. The SPEC-048 doc was updated but the docker-compose comment was not.
   - This is a doc consistency issue, not a functional bug.

### Suggestions

1. **`audio-transcriptions.ts` — Fail-open on LLM correction is intentional but worth documenting**
   - Line ~86: `return text; // fail open — retorna pré-processado`
   - This is correct per spec (PT-BR correction is best-effort), but the 30s timeout on Ollama could mask longer outages. Consider adding a metric or log when falling back.

2. **`smoke-multimodal-stack.sh` — Vision smoke test is now a warning-only**
   - The vision test (`llava-phi3` cold start) can take 15-30s. The updated test tolerates non-200 codes and demotes failure to a warning. This is pragmatic, but the test is now less useful as a regression detector.
   - Suggestion: Split into a separate `--slow` test mode that enforces vision, vs fast mode that only checks routing.

3. **`setup-vl-model.sh` — Interactive confirmation in non-interactive context**
   - Line ~165: `read -r -p "Usar '$MODEL'...? [s/N] " confirm`
   - This script could be run in a CI/CD pipeline (e.g., `/ship` or `/turbo` loops). The interactive prompt would hang indefinitely.
   - Suggestion: Accept a `--yes` flag to skip confirmation, or check `if [ -t 0 ]` (stdin is a terminal).

4. **`setup-whisper-medium-ptbr.sh` — PID file not cleaned up on restart**
   - The script writes `"/tmp/whisper-medium-ptbr.pid"` but does not remove it when restarting an existing server.
   - Suggestion: On re-run, kill the old PID from the file before starting a new one.

5. **`audio-transcriptions.ts` — No request ID / correlation ID**
   - The route writes errors to `stderr` as `[stt] error: ${msg}` but has no request-scoped ID. Correlating logs across gateway → wav2vec2 → Ollama is difficult.
   - Suggestion: Add a `crypto.randomUUID()` at request start and include it in all log lines.
