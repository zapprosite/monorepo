# Plan 03-04 - Intake integration with dry-run, index and pending review reasons

<objective>
Connect finder/downloader output to the canonical intake gate so candidate PDFs can be dry-run validated, indexed when accepted, or recorded in pending review with explicit reasons.
</objective>

## Objective

Implement a traceable acquisition flow:

```text
ranked candidate -> validated PDF -> hvac_add_manual.py --dry-run -> accepted/indexed or pending_review.jsonl
```

1. Define an acquisition result payload with:
   - `brand`
   - `model`
   - `candidate_url`
   - `download_status`
   - `intake_status`
   - `pending_review_reason`
   - `pdf_path`
   - `sha256`
2. Add helper to append JSONL records to `pending_review.jsonl`.
3. Include rejection reasons from downloader and intake without losing source URL/domain.
4. Keep docs/user-facing strings in PT-BR; code identifiers in English.

<acceptance_criteria>
- Failed download writes a pending review record with reason.
- Intake rejection writes a pending review record with reason.
- Accepted dry-run does not index unless `--index` is explicitly requested.
</acceptance_criteria>
</task>

<task type="auto" tdd="true">
<name>Task 2: Implement dry-run/index orchestration</name>
<read_first>
scripts/hvac-rag/hvac_add_manual.py
scripts/hvac-rag/hvac_manual_finder.py
scripts/hvac-rag/hvac_manual_downloader.py
</read_first>
<files>scripts/hvac-rag/hvac_manual_finder.py, tests/hvac-rag/test_manual_acquisition_flow.py</files>

1. Add CLI mode to process one ranked candidate or JSONL candidate list.
2. Run downloader first; stop on non-accepted/duplicate statuses with structured output.
3. Run `hvac_add_manual.py --dry-run` for validated PDFs.
4. If `--index` is present and dry-run is accepted, run `hvac_add_manual.py --index`.
5. Capture subprocess exit code and safe stdout/stderr snippets without printing secrets.

<acceptance_criteria>
- `--dry-run` path does not mutate the Qdrant/index manifests.
- `--index` is the only path that indexes accepted PDFs.
- Subprocess failures become structured `intake_status="error"` results.
</acceptance_criteria>
</task>

- Every candidate ends in exactly one explicit state: downloaded, duplicate, rejected, accepted_dry_run, indexed, or pending_review.
- `pending_review.jsonl` contains enough reason/source metadata for manual triage.
